# Anticipated results

The results described here come from running the commands in
`tutorials/full_ecModel/protocol.m`, unless otherwise indicated. The exact
output may differ somewhat numerically depending on updates at the various
databases and on the solver used.

## Stage 0 and Stage 1

- **Step 1.** The adapter folder is initiated at the user-specified location,
  containing empty subfolders `code`, `data`, `models` and `output`, plus a
  template model adapter file starting with the model name and suffixed with
  `Adapter.m`.
- **Step 2.** Typically the starting GEM file in the `models` subfolder of the
  adapter folder.
- **Steps 3-7.** The model adapter file is populated with organism- and
  model-specific parameters. If no appropriate KEGG or UniProt parameters could
  be identified, custom `data/uniprotConversion.tsv`, `data/kegg.tsv` and/or
  `data/uniprot.tsv` files are constructed.
- **Step 8.** The MATLAB instance is primed for a particular ecModel by setting
  the relevant `modelAdapter` as default.
- **Step 9.** The conventional GEM is loaded into the MATLAB workspace in a
  structure named `model`. Simulations can be performed as with any GEM.
- **Steps 10-11.** Enzyme data are downloaded from UniProt and KEGG and stored
  in `data/uniprot.tsv` and `data/kegg.tsv`. An empty ecModel is generated and
  named `ecModel` in the workspace.

    A `noUniprot` structure may list genes with no UniProt match. In the
    `light_ecModel` tutorial, `noUniprot` contains six genes that could not be
    mapped. Because this is a low number, it is probably futile to change
    `obj.params.uniprot.ID`; instead, manually curate the gene identifiers,
    `uniprotConversion.tsv`, or add the relevant entries to the UniProt database
    file. Regardless, an empty ecModel is generated with enzymes for all matching
    genes, and behaves similarly to the conventional GEM it was derived from.

- **Steps 12-13.** Enzyme complex data is retrieved and stored in
  `data/ComplexPortal.json`, then integrated into the ecModel, modifying
  `ecModel.ec.rxnEnzMat` with stoichiometric coefficients. In the `full_ecModel`
  tutorial the command window displays:

    ```
    A total of 206 complex have full match, and 17 proposed.
    ```

    As an example of `proposedComplex`, reaction `r_0505_EXP_2` could be matched
    with complex CPX-1293, the mitochondrial 2-oxoglutarate dehydrogenase
    complex. In the ecModel this reaction is annotated with proteins P20967
    (KGD1, subunit E1), P19262 (KGD2, subunit E2) and P09624 (LPD1, subunit E3),
    while CPX-1293 also includes P19955 (YMR31), known as 37S ribosomal protein.
    Literature confirms that P19955, also termed KGD4 or subunit E4, is an
    essential subunit for a stable complex. On that basis, the yeast-GEM should
    be curated with the updated gene association for reaction `r_0505`, which
    would give a 100% match.

- **Step 14.** The ecModel is stored in the chosen format (SBML or YAML) in the
  `models` subfolder. If stored in SBML, the file can be loaded by any other
  software that accepts SBML.

## Stage 2

- **Steps 15-19.** The `ecModel.ec.eccodes` field is curated to contain EC
  numbers. $k_{cat}$ data are collected from BRENDA and saved as a `kcatList`
  (for example `kcatList_fuzzy`) with the following fields:

    - `source`: `brenda` when generated by `fuzzyKcatMatching`, or `dlkcat` when
      generated by `readDLKcatOutput`.
    - `rxns`: reaction identifiers matching `ecModel.ec.rxns`.
    - `substrates`: names of all substrates used to match to BRENDA.
    - `kcats`: proposed $k_{cat}$ values from fuzzy matching.
    - `eccodes`: EC numbers used to match to BRENDA.
    - `wildcardLvl`: level of EC wildcard needed to find a match
      (0: w.x.y.z, 1: w.x.y.-, 2: w.x.-.-, 3: w.-.-.-).
    - `origin`: level of specificity needed to find a match
      (1: correct organism, correct substrate; 2: closest related organism,
      correct substrate; 3: correct organism, any substrate; 4: closest related
      organism, any substrate; 5: correct organism, specific activity;
      6: closest related organism, specific activity).

- **Steps 20-25.** SMILES matches from `findMetSmiles` are stored in
  `data/smilesDB.tsv`, and `ecModel.metSmiles` is filled. The command window
  displays:

    ```
    Check for local SMILES database... done.
    SMILES could be found for 64% of the unique metabolite names.
    ```

    An input file for DLKcat is stored in `data/DLKcat.tsv`. Once `runDLKcat`
    runs, predicted values are appended to the same file. The resulting
    `kcatList` is similar to the BRENDA one but lacks the `eccodes`,
    `wildcardLvl` and `origin` fields, and instead includes a `genes` field.

- **Step 26.** The merged `kcatList` is similar to those above, plus a
  `kcatSource` field indicating the source for each value individually.
- **Step 27.** The `ecModel.ec.kcat` field is populated with selected values,
  with `ecModel.ec.source` indicating where each is derived from.
- **Step 28.** The `ecModel.ec.kcat` field is further populated with custom
  values, overwriting existing values.
- **Step 29.** The `ecModel.ec.kcat` field is updated so that all isozymic
  reactions have values; `ecModel.ec.source` for those reactions indicates
  `isozymes`. `ecModel.S` remains unchanged.
- **Step 30.** The `ecModel.ec` structure gains a `standard` pseudo-enzyme of MW
  44,898 Da, and reactions in `rxnsMissingGPR` and `rxnsNoKcat` receive the
  standard $k_{cat}$ value of 12.3 s^-1.
- **Step 31.** The values added or changed during Steps 27-30 are applied to
  `ecModel.S`. The resulting stoichiometric coefficients of the enzyme
  pseudo-metabolites are defined as:

    $$ \frac{MW}{k_{cat} \times 3600} \times (\text{complex stoichiometry}) $$

    where the 3,600 factor converts $k_{cat}$ from s^-1 to h^-1 to reconcile
    with the unit of metabolic rates. Each enzymatic reaction is assigned one
    $k_{cat}$ value, which for complexes can be the highest predicted value
    across subunits. For example, if reaction X to Y is catalyzed by a complex
    of subunits A, B and C at a ratio 1:1:2, all with MW 27,000 Da and highest
    predicted $k_{cat}$ of 30 s^-1, the reaction becomes
    X + 0.25 A + 0.25 B + 0.5 C to Y.

- **Step 32.** The protein pool exchange reaction is constrained by the
  calculated maximum enzyme usage, implemented as a negative lower bound. If
  $P_{tot}$ is 0.5 g/gDCW, f is 0.5 and sigma is 0.5, then the `ecModel.lb`
  entry for `prot_pool_exchange` is -125 mg/gDCW.

## Stage 3

- **Steps 33-36.** The growth rate immediately after applying the constraints
  appears as:

    ```
    Growth rate: 0.107203 /hour
    ```

    Because nutrient uptake is not limiting, this is the maximum the ecModel can
    reach. Being substantially below 0.41 h^-1 indicates the need for $k_{cat}$
    tuning.

- **Steps 37-38.** The maximum growth rate of the starting GEM, without
  constraints on carbon uptake:

    ```
    Growth rate: 19.718033 /hour
    ```

    This is not biologically realistic (unconstrained carbon uptake), but it
    signifies that the conventional GEM stoichiometry can reach 0.41 h^-1, so the
    problem in Step 35 is due to the enzyme constraints.

- **Steps 39-41.** Setting `prot_pool_exchange` to -1,000 mg/gDCW implies the
  whole cell is 100% protein. While unrealistic, this often avoids
  overconstraining. The simulation shows:

    ```
    Protein pool usage is: 446 mg/gDCW
    ```

    With f = 0.5, this implies the cell would consist of 89.2% protein. Assuming
    full enzyme saturation, a more realistic lower sigma would push the simulated
    total protein content above 100%, which is why simply releasing the protein
    pool constraint is not recommended.

- **Steps 43-44.** Iterative tuning increases the growth rate toward 0.41 h^-1:

    ```
    Iteration 1: Growth: 0.11421
    Iteration 2: Growth: 0.18016
    Iteration 3: Growth: 0.23412
    Iteration 4: Growth: 0.27495
    Iteration 5: Growth: 0.30635
    Iteration 6: Growth: 0.33961
    Iteration 7: Growth: 0.3718
    Iteration 8: Growth: 0.39141
    Iteration 9: Growth: 0.41193
    ```

    The selected `ecModel.ec.kcat` entries are listed in `tunedKcats`
    (see [Example sensitivityTuning output](#example-sensitivitytuning-output)).
    The `ecModel.ec.source` still refers to the original source, so it is
    recommended to inspect `tunedKcats` and include the required adjustments in
    Step 28.

- **Steps 45-52.** For reaction `r_0079`, the `kcatList` indicates `wildcardLvl`
  0 and origin 4. Querying BRENDA and converting the reported specific activity
  yields a new value of 5.34 s^-1, matching the value found by
  `sensitivityTuning`. The value is applied directly with `setKcatForReactions`,
  but best practice is to document curated values in `customKcats.tsv`.

## Stage 4

- **Steps 53-57.** A `protData` structure holds the proteomics data from
  `data/proteomics.tsv`. It has two fields: `abundances` (columns are datasets,
  rows are proteins) and `uniprotIDs`. The `ecModel.ec.concs` field is populated
  by `fillEnzConcs`. After `constrainEnzConcs`, enzymes with concentrations no
  longer draw from `prot_pool` but are constrained by setting `ecModel.lb` to
  minus their concentration.
- **Step 58.** A `fluxData` structure holds reaction flux and total protein data
  with fields `conds`, `Ptot`, `grRate`, `exchFluxes`, `exchMets` and
  `exchRxnIDs`. The recalculated `prot_pool` constraint is applied by
  `updateProtPool`.
- **Steps 59-63.** The relevant `ecModel.lb` and `ecModel.ub` entries are
  updated. The command window reports:

    ```
    Growth rate that is reached: 0.004578 /hour
    ```

    This is far less than the intended 0.1 h^-1 specified in `fluxData.grRate`.

- **Steps 64-65.** `flexibilizeEnzConcs` reports its progress, detailing each
  flexibilized enzyme and the new growth rate. In the `full_ecModel` tutorial,
  flexibilization alone is not sufficient to reach 0.1 h^-1 (abridged):

    ```
    Protein P37299 LB adjusted. Grow: 0.0097884
    Protein Q03028 LB adjusted. Grow: 0.012172
    Protein P24521 LB adjusted. Grow: 0.012897
    Protein P00128 LB adjusted. Grow: 0.013333
    [...]
    Protein P32473 LB adjusted. Grow: 0.088025
    Protein P07285 LB adjusted. Grow: 0.088895
    Protein P12695 LB adjusted. Grow: 0.088899
    Protein P36148 LB adjusted. Grow: 0.0889
    No (more) limiting enzymes have been found. Attempt to increase protein
    pool exchange...
    Protein pool exchange was also not limiting. Inability to reach growth
    rate is not related to enzyme constraints. Maximum growth rate is 0.0889.
    ```

    Simulating the same condition in the conventional GEM confirms the cause:

    ```matlab
    model = constrainFluxData(model, fluxData);
    sol = solveLP(model);
    fprintf('Growth rate that is reached: %f /hour\n', abs(sol.f))
    ```

    Output:

    ```
    Growth rate that is reached: 0.0889 /hour
    ```

    This confirms the growth rate is overconstrained by exchange rates, possibly
    because of measurement errors, rather than enzyme constraints.

## Stage 5

- **Steps 66-68.** Simulating the Crabtree effect demonstrates the effect of the
  total protein constraint. The ecModel with infinite protein pool (mimicking a
  conventional GEM) cannot predict the shift to fermentation at high growth
  rate. The ecModel before sensitivity tuning is also unable to simulate higher
  growth rates if some $k_{cat}$ values are too low. Only 13 $k_{cat}$ values had
  to be tuned for realistic maximum growth rates. The ecModel with tuned values
  exhibits the Crabtree effect: yeast switches from respiration (oxygen
  exchange) to fermentation (ethanol exchange) when the protein pool becomes
  fully used, constraining the energetically more efficient respiratory flux
  distribution. In the simulation the protein pool becomes limiting at a growth
  rate that is a little too low (the simulated glucose uptake slope increases
  after 0.2 h^-1, while experimentally this happens from around 0.28 h^-1),
  suggesting one enzyme dominates the constraint too early. Inspecting the top
  used enzymes (Step 71) can be informative.
- **Steps 69-70.** Constraining to at least 99% of the maximum growth rate while
  minimizing the protein pool usage:

    ```
    Growth rate that is reached: 0.4121 /hour
    Minimum protein pool usage: 123.75 mg/gDCW
    ```

- **Step 71.** Constraining to at least 0.25 h^-1 while minimizing glucose
  exchange. The `usageData` structure holds the absolute and capacity usages,
  while `usageReport` organizes them in `topAbsUsage` and `highCapUsage`.
  Because this ecModel has concentrations in `ecModel.ec.concs`, `highCapUsage`
  is empty. The top ten enzyme usages are shown in the
  [Top ten enzyme usages](#top-ten-enzyme-usages-at-growth-rate-025-h-1) table.
- **Step 72.** The flux distribution is mapped back so `mappedFlux` is a vector
  with the same length (4,063 entries) as `model.rxns`. The `enzUsageFlux`
  vector holds the enzyme usage fluxes not present in conventional GEMs, and
  `usageEnz` holds the corresponding protein identifiers.
- **Steps 73-75.** The `minFlux` and `maxFlux` matrices have three columns each,
  one per model, ordered as in `model.rxns`. The flux variability visualization
  (stored in `output/ecFVA.pdf`) shows that the median flux variability of the
  ecModel is more than four-fold lower than the conventional GEM. The
  conventional GEM has more than 15% of reactions with very high flux variability
  (around 1,000 mmol/gDCWh), which is absent from the ecModels. Median
  variabilities reported were GEM 3.104, ecModel without proteomics 0.35479, and
  ecModel with proteomics 0.003253.
- **Steps 76-77.** A light and full ecModel of yeast-GEM are made, populated only
  with `fuzzyKcatMatching` values. The command window reports:

    ```
    Comparison of duration light vs. full ecModel
    ecModel reconstruction: 95% (39 vs 41 seconds)
    FBA: 64% (0.57 vs 0.89 seconds)
    Mapping fluxes: 64% (0.103 vs 0.161 seconds)
    Growth rate that is reached: 0.0252 vs 0.0252
    ```

    Reconstruction does not differ much (both require the time-demanding
    `fuzzyKcatMatching`), but FBA and flux mapping are substantially faster in
    light ecModels. The light ecModel is particularly useful when many
    simulations are performed, such as extensive ecFVA or many single-cell
    ecModels. If proteomics integration is needed, a full ecModel is justified.
    The flux distributions are near identical between the two ecModels; the only
    reactions deviating by more than 0.1% are in lipid metabolism. Lipid
    metabolism in yeast-GEM follows the SLIMEr formulation, where reactions for
    each potential fatty acyl-chain configuration are explicit and catalyzed by
    the same enzyme, introducing flexibility in the paths through lipid
    metabolism that all reach the same overall acyl-chain distribution. This
    flexibility is inherited from the conventional GEM and is not affected by the
    enzyme constraints.
- **Steps 78-79.** The resulting `ecHT29` is based on the generic ecModel and
  contains all its enzyme and $k_{cat}$ information for the reactions and genes
  present in the contextualized GEM HT29. Comparing maximum growth rates shows
  the impact of both enzyme constraints and contextualization:

    ```
    Growth rate in HT29-GEM: 149.97 /hour.
    Growth rate in ecHuman-GEM: 0.122 /hour.
    Growth rate in ecHT29-GEM: 0.086 /hour.
    ```
